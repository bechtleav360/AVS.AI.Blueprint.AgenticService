# Goal
Create deterministic tools that your agent can call during runtime to gather data or perform domain-specific checks.

## When to implement tools
- **Required**: If your agent needs to call external services (REST APIs, databases) or reusable utility functions during execution.
- **Optional**: If all logic can be handled with pure functions inside `logic.py`, you may skip tools, but most production agents use at least one tool for data enrichment.

## 1. Understand the tool contract

Tools live in `agent/src/custom/agent/tools.py`. Each tool is typically a method on a `Tools` or `CustomTools` class and should:

- Accept a `RunContext[ProcessingContext]` object (provides config, tracing, logger).
- Accept a typed input model (`Pydantic` model defined in `agent/src/custom/models/`).
- Return a typed `Pydantic` model or primitive value that the agent runtime can consume.

Example skeleton:

```python
class CustomTools:
    def __init__(self, settings: Settings) -> None:
        self.settings = settings

    async def fetch_asset_details(
        self,
        ctx: RunContext[ProcessingContext],
        payload: AssetLookupInput,
    ) -> AssetMetadata:
        ...
```

## 2. Define input and output models

Create or update models under `agent/src/custom/models/`:

- **Input model** (e.g., `AssetLookupInput`) with fields like `asset_id: str`.
- **Output model** (e.g., `AssetMetadata`) that matches the external service schema.

Add docstrings and type hints so junior developers understand the purpose of each field. Keep DTOs separate from event envelopes to avoid confusion.

## 3. Implement the tool logic

Follow these best practices:

1. **Use `httpx` for HTTP calls**
   ```python
   async with httpx.AsyncClient(timeout=self.settings.http_timeout) as client:
       response = await client.get(...)
   response.raise_for_status()
   data = response.json()
   return AssetMetadata.model_validate(data)
   ```

2. **Propagate tracing**: Include headers from `ctx.traceparent` if available.

3. **Handle errors gracefully**: Translate network errors or validation issues into domain-specific exceptions so the runtime can decide whether to retry or fail fast.

4. **Log meaningful context**: Use `ctx.logger` to log asset IDs, tenant IDs, and reasons for retries. Never log secrets.

## 4. Register tools with the runtime

Update `agent/src/custom/agent/runtime.py` so `_get_tools()` returns your tool methods. Example:

```python
def _get_tools(self) -> list[Callable[..., Any]]:
    tools = CustomTools(settings=self.settings)
    return [
        tools.fetch_asset_details,
        tools.check_backup_window,
    ]
```

Use descriptive function names so logs clearly indicate which tool ran.

## 5. Test tools in isolation

Add unit tests under `agent/tests/unit/`:

- Mock external dependencies with `respx` or `pytest-httpx`.
- Assert that `httpx` is called with the right URL, headers, and timeouts.
- Verify retries or fallback behavior if applicable.

Example test snippet:

```python
@pytest.mark.asyncio
async def test_fetch_asset_details_success(httpx_mock):
    httpx_mock.get("https://gateway/assets/a-123", json={"asset_id": "a-123"})

    tools = CustomTools(settings=settings)
    result = await tools.fetch_asset_details(ctx, AssetLookupInput(asset_id="a-123"))

    assert result.asset_id == "a-123"
```

## 6. Checklist
- [ ] Input/output models defined with docstrings
- [ ] Tool methods implemented with timeouts and error handling
- [ ] Tools registered in `runtime.py`
- [ ] Unit tests cover success and failure paths
- [ ] Logs avoid secrets but include enough context for debugging
