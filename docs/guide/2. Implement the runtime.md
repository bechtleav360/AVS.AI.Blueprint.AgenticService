## Goal
Assemble the custom agent runtime so it can process thin and fat events, call deterministic tools, and produce structured results.

## Prerequisites
- Complete the steps in `docs/guide/1. Initialize the agent.md` (fork, environment, roadmap).
- Familiarity with the directories under `agent/src/custom/`:
  - `agent/src/custom/agent/runtime.py`
  - `agent/src/custom/agent/logic.py`
  - `agent/src/custom/agent/tools.py`
  - `agent/src/custom/models/`

## 1. Understand the runtime template

The base project wires the runtime through `agent/src/main.py`, which imports `CustomAgentRuntime` from `agent/src/custom/agent/runtime.py`. The runtime is a thin wrapper around the Pydantic AI agent that:

- Loads prompts from `agent/src/custom/prompts/`.
- Registers tools (deterministic functions) that can be invoked by the LLM.
- Exposes a typed `CustomAgentOutput` so the HTTP responses and result events are structured.

Read through `agent/src/custom/agent/runtime.py` first. Pay attention to methods like `_get_tools()`, `_get_prompt_name()`, and `run()`—they are the extension points.

## 2. Define or refine models

Check `agent/src/custom/models/`:

- `context.py` (or similarly named file) provides the **processing context**. Add any fields you need (tenant IDs, correlation IDs, flags) but keep them serializable.
- `resource.py` or other input models describe the data you pass to the runtime.
- `result.py` (often `CustomAgentOutput`) describes the agent response.

Whenever you add fields, ensure their types are explicit and documented via docstrings. This makes the autogenerated OpenAPI docs clearer.

## 3. Wire deterministic logic (`logic.py`)

`agent/src/custom/agent/logic.py` should contain the core decision-making functions for your domain. Example patterns:

- `def score_backup_signals(asset: AssetMetadata) -> Score`: calculates a confidence score from metadata.
- `def pick_next_action(evidence: list[str]) -> AgentDecision`: decides whether to run another tool or stop.

These functions can call external services when necessary, but you must wrap those calls so they can be mocked in unit tests (for example, inject the client or use dependency-overridable helpers). This keeps the code testable while still allowing real integrations.

## 4. Implement tools (`tools.py`)

Tools bridge the runtime with side effects (HTTP calls, database queries). Each tool should:

1. Accept a `RunContext` and typed payload.
2. Handle retries/timeouts if necessary (start simple—wrap external HTTP calls with `httpx` and a timeout).
3. Return typed models defined in the `models/` package.

When a tool becomes complex, extract helper functions so juniors can follow the flow. Include docstrings that explain what each tool does and when to use it.

## 5. Customize the runtime (`runtime.py`)

Open `agent/src/custom/agent/runtime.py` and:

1. **Register tools**
   ```python
   def _get_tools(self) -> list[Callable[..., Any]]:
       tools = CustomTools(settings=self.settings)
       return [
           tools.fetch_asset_details,
           tools.check_backup_window,
       ]
   ```

2. **Select the system prompt**
   ```python
   def _get_prompt_name(self) -> str:
       return "system"  # matches `agent/src/custom/prompts/system.prompt`
   ```

3. **Expose the result type**
   ```python
   def _get_result_type(self) -> type[BaseModel]:
       return CustomAgentOutput
   ```

4. **Implement `run()`** – this is where you compose context, call the agent, and post-process the result. A typical flow:
   ```python
   async def run(self, resource: ResourceInput, context: ProcessingContext) -> CustomAgentOutput:
       agent = self._build_agent()
       result = await agent.run(
           prompt_name=self._get_prompt_name(),
           tools=self._get_tools(),
           context=context,
           input_data=resource,
       )
       return self._post_process(result)
   ```

Keep the runtime focused on orchestration. Heavy business logic should remain in `logic.py`.

## 6. Connect with the chain of responsibility

`agent/src/custom/agent/decision.py` (or similarly named file) decides whether your runtime should run for an incoming event. Ensure the runtime exposes functions that the CoR can call, such as `run_check(asset: AssetMetadata)`.

When handling **thin events**:

- Fetch additional data via a tool (`tools.fetch_asset_details`).
- Merge the fetched data with the event payload before calling `run()`.

For **fat events**:

- Pass the asset payload directly to the runtime without extra calls.

## 7. Test locally

Run the unit tests to verify your runtime wiring:

```bash
pytest agent/tests/unit/test_runtime.py -vv
```

If the file does not exist, create one and cover:
- Prompt selection
- Tool registration
- Result type enforcement

- [ ] Models updated and documented (`agent/src/custom/models/`)

- [ ] Tools registered with clear docstrings (`agent/src/custom/agent/tools.py`)
- [ ] Runtime orchestrates prompts, tools, and results (`agent/src/custom/agent/runtime.py`)
- [ ] Chain of responsibility integrates with runtime
