# Goal
Keep your fork of the blueprint current, apply upstream improvements safely, and communicate changes back to the community.

## 1. Sync with upstream

1. Fetch the latest changes:
   ```bash
   git fetch upstream
   ```
2. Create a branch dedicated to the sync (avoids polluting your current feature work):
   ```bash
   git checkout main
   git pull origin main
   git checkout -b chore/sync-upstream-YYYYMMDD
   ```
3. Merge or rebase onto your fork:
   ```bash
   git merge upstream/main
   # or
   git rebase upstream/main
   ```
4. Resolve conflicts carefully. When in doubt, favor upstream defaults and re-apply your customizations later.

## 2. Validate after merging

1. Reinstall dependencies if `requirements.txt` or `pyproject.toml` changed.
2. Run the project-wide quality gates:
   ```bash
   make lint
   make test
   ```
3. If the merge introduced new configuration files (e.g., new dynaconf settings), update your local `.env` so tests still pass.

## 3. Review blueprint docs

1. Compare `docs/architecture.md`, and `docs/guide/` from upstream with your local copies.
2. Update your roadmap or implementation notes to reflect any new requirements or deprecations.
3. If upstream added new steps (e.g., additional Dapr components), integrate them into your deployment and testing workflows.

## 4. Communicate changes

1. Summarize the sync in your project changelog (`docs/changelog.md`) or via your team’s communication channels.
2. If you discovered breaking changes or bugs, open an issue in the upstream repository with clear reproduction steps.
3. When your fork introduces improvements that may help others, prepare a pull request back to upstream with:
   - Summary of the enhancement.
   - Testing evidence (commands run, environments).
   - Follow-up tasks if additional work is required.

## 5. Checklist
- [ ] Upstream changes fetched and merged/rebased
- [ ] Conflicts resolved and dependencies aligned
- [ ] Lint and test suites pass after the update
- [ ] Documentation and roadmap refreshed to match new blueprint guidance
- [ ] Findings communicated to the team and/or upstream maintainers
