# Goal
Prepare a clean working environment for building your custom agent without disrupting the shared blueprint.

## Prerequisites
- Git ≥ 2.40 installed and configured with your name and email (`git config --global user.name "Your Name"`).
- Python 3.13 with `pip`.
- Docker + Docker Compose (for local runtime parity).
- Access to the upstream `av360/Agents_Blueprint` repository.

> **Tip:** Junior developers often get blocked by missing tools. If a command fails, read the error carefully—it usually hints at the missing dependency.

## 1. Fork and clone the repository
1. Open the Azure DevOps web interface and navigate to the shared project: [`Agents_Blueprint`](https://dev.azure.com/av360/Bechtle-Index-of-Sovereignty/_git/Agents_Blueprint).
2. Select **Fork** and choose **Same project** so a new repository is created under the same team (for example `MyCustomAgent`).
3. Confirm the fork and wait until Azure DevOps finishes provisioning the new repository.
4. Copy the HTTPS clone URL from the forked repository page (for example `https://dev.azure.com/av360/Bechtle-Index-of-Sovereignty/_git/MyCustomAgent`).
5. Clone your fork locally and enter the project directory:
   ```bash
   git clone https://dev.azure.com/av360/Bechtle-Index-of-Sovereignty/_git/MyCustomAgent
   cd MyCustomAgent
   ```
6. Add the original repository as an `upstream` remote so you can sync changes later:
   ```bash
   git remote add upstream https://dev.azure.com/av360/Bechtle-Index-of-Sovereignty/_git/Agents_Blueprint
   git fetch upstream
   ```

## 2. Follow GitFlow branching
We use GitFlow to keep work organized:

- `main`: Always deployable. Only tagged releases are merged here.
- `develop`: Integration branch for completed features. Treat this as your staging baseline.
- `feature/*`: Short-lived branches created from `develop` for individual tasks.
- `release/*`: Temporary branches to prepare a production release; created from `develop`, merged into both `main` and `develop` when done.

### Creating and updating branches

```bash
# make sure you start from develop for feature work
git checkout develop
git pull origin develop

# create a new feature branch
git checkout -b feature/add-backup-checker
git push -u origin feature/add-backup-checker

# start a release branch when you're ready to harden for prod
git checkout develop
git pull origin develop
git checkout -b release/2025.10.0
git push -u origin release/2025.10.0
```

### Suggested branch names
- **Feature ideas:** `feature/add-backup-checker`, `feature/enable-fat-events`, `feature/improve-idempotency`
- **Release ideas:** `release/2025.10.0`, `release/2025.11.0`, `release/hotfix-otlp-config`

Always open pull requests back into `develop`. When a release branch is ready, merge it into `main` (with a tag) and back into `develop` to capture hotfixes.

## 3. Set up your Python environment with `uv`
[`uv`](https://github.com/astral-sh/uv) is a fast Python package manager that handles virtual environments and dependency resolution.

1. Install `uv` once (if you do not already have it):
   ```bash
   curl -LsSf https://astral.sh/uv/install.sh | sh
   ```
   Restart your shell so the `uv` command is available.
2. Create and activate a project-scoped virtual environment:
   ```bash
   uv venv .venv
   source .venv/bin/activate
   ```
3. Install dependencies for both the base framework and the custom agent:
   ```bash
   uv pip install -r base/requirements.txt
   uv pip install -r agent/requirements.txt
   ```
4. Enable the formatting hooks that keep the repo consistent:
   ```bash
   uv pip install pre-commit
   pre-commit install
   ```

Keep the virtual environment activated whenever you work on the project (`source .venv/bin/activate`).

## 4. Configure local settings
1. Copy the provided environment template:
   ```bash
   cp agent/.env.example agent/.env
   ```
2. Fill in values such as `DATA_GATEWAY_BASE_URL` or leave placeholders if you do not know them yet. Never commit secrets—store them in `.env` only.
3. Review `settings.toml` under the `agent/` and `base/` directories to understand default configuration values.

## 5. Update documentation for your agent
Update both `README.md` (root) and `agent/README.md` so reviewers understand your intent.

- **Motivation:** What business problem does the agent solve?
- **Definition of done:** List success criteria (e.g., "Publishes thin results to `assets.results.backup-check`" or "Integration tests cover thin vs. fat flows").
- **Roadmap preview:** Outline upcoming milestones. Use the checklist below as a starting point.

Consider adding a high-level architecture diagram in `docs/assets/` to communicate the thin/fat event flow.

## 6. Plan your roadmap
Document your path forward in `docs/roadmap.md`:

- Milestones for Weeks 1–2 (MVP) and Weeks 3–4 (hardening).
- Risks and assumptions (e.g., "Need access to Data Gateway sandbox").
- Links to tickets or tasks.

This roadmap will feed future status updates and blueprint changes.

## 7. Verify everything works locally
Run the baseline checks before you start modifying code:

```bash
make lint
make test
docker compose -f docker-compose.yml config
```

Fix any issues immediately; otherwise future debugging becomes harder.

## 8. Quick checklist

- [ ] Feature branch pushed
- [ ] Virtual environment activated
- [ ] Dependencies + pre-commit installed
- [ ] README and roadmap updated
- [ ] Baseline lint/tests pass

You are now ready to implement the runtime.
