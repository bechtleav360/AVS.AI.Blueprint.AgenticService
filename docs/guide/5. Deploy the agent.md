# Goal
Ship your agent to a target environment (staging, then production) with confidence and repeatability.

## Prerequisites
- All tests pass locally (`make test`).
- Docker image builds successfully (`docker compose build`).
- Access to the target infrastructure (Kubernetes cluster, VM, or container platform).
- Credentials stored securely (secret manager, GitHub Actions secrets, Azure Key Vault, etc.).

## 1. Prepare deployment artifacts

1. **Container image**
   ```bash
   docker build -t <registry>/asset-backup-checker:latest -f agent/Dockerfile .
   docker push <registry>/asset-backup-checker:latest
   ```
2. **Configuration**
   - `agent/settings.toml`: Verify environment-dependent values (e.g., `DATA_GATEWAY_BASE_URL`).
   - `.env`: Provide runtime secrets at deploy time, not in version control.
3. **Helm chart / Compose / Manifests**
   - If deploying to Kubernetes, create or update Helm values referencing the new image tag.
   - For VM deployments, prepare a systemd unit or Docker Compose bundle.

## 2. Staging deployment checklist

1. Allocate infrastructure (namespace, RabbitMQ instance, Data Gateway credentials).
2. Apply Dapr components from `custom/dapr/components/` and ensure secrets are resolved.
3. Deploy the service (Helm, kubectl apply, or Docker Compose).
4. Verify the following endpoints:
   - `GET /actuators/health`
   - `GET /actuators/livez`
   - `GET /actuators/readyz`
5. Publish a thin and fat test event via Dapr CLI to ensure the agent processes messages end to end.

Record any deviations or manual steps in `docs/deployment-guide.md` so production deploys are smoother.

## 3. Observability setup
- Ensure OpenTelemetry exporter endpoints are reachable (check `OTEL_EXPORTER_OTLP_ENDPOINT`).
- Hook tracing dashboards (Jaeger, Azure Monitor) to confirm spans for thin/fat flows appear.
- Configure logging aggregation (e.g., ELK, Stackdriver) and filter out secrets or PII.

## 4. Production readiness review
- **Security**: Secrets rotated, TLS enabled, ingress rules restricted.
- **Performance**: Load-test the agent path using realistic event volumes.
- **Failover**: Confirm DLQ policies are enforced; practice draining DLQ messages.
- **Runbooks**: Update `docs/troubleshooting.md` with common failure modes and remediation steps.

## 5. Cutover plan
1. Announce deployment window to stakeholders.
2. Tag the release in Git (`git tag v1.0.0 && git push --tags`).
3. Deploy to production using the same automation scripts used for staging.
4. Monitor logs, traces, and alerting channels for at least one event cycle.

If issues arise, roll back by redeploying the previous image tag and document the incident.

## 6. Automation tips
- Integrate image builds and Helm packaging into your CI pipeline (see `agent/azure-pipelines.yml`).
- Use environments in GitHub Actions or Azure DevOps to require manual approval before production.
- Store configuration in Dynaconf-compatible secret stores for consistent loading across environments.

## 7. Checklist
- [ ] Image built and pushed to registry
- [ ] Configuration values reviewed for target environment
- [ ] Staging deployment verified with health checks and test events
- [ ] Observability hooks confirmed (traces, logs, metrics)
- [ ] Production readiness checklist complete
- [ ] Release tagged and communicated
